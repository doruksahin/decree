## Methodology

- Per-query retrieval scores are passed to `ir_measures.calc_aggregate` /
  `iter_calc` (MIT-licensed canonical IR-metrics library) — no hand-rolled
  metric code.
- 95% confidence intervals computed via `scipy.stats.bootstrap`
  (`method="percentile"`, `n_resamples=200`,
  `random_state=42` for reproducibility).
- Abstention queries (`relevant: []`) are excluded from metric computation
  (avoids divide-by-zero / 0/0 noise) but are still reported under
  per-query breakdown.
- K values: 1, 3, 5, 10.
- Query set: hand-authored against the `decree` corpus,
  shipped in-tree at `eval/queries.yaml` for visibility.

## Limitations

1. **Small corpus.** The decree corpus is ~17 documents in v1, far below
   the size at which retrieval-quality numbers are statistically informative
   in absolute terms. Numbers here are useful for *ablation* (relative
   delta against the frozen `keyword-v1` baseline), not for absolute claims
   about quality.
2. **Single-author labels.** The query set is hand-authored by one person.
   Inter-annotator agreement is therefore undefined; latent labeler bias
   (over-weighting queries where the keyword baseline already wins) is
   plausible.
3. **No external corpus.** v1 evaluates against decree-only documents —
   results do not generalise to other corpora. A future SPEC will add an
   open-source ADR corpus (CNCF, C4 model project, etc.).
4. **Binary relevance dominant.** Most queries use binary relevance even
   though the schema supports graded; graded judgments would change
   nDCG@10 meaningfully on borderline cases.
5. **Bootstrap requires N≥2.** Single-relevant-doc queries with empty runs
   still contribute to the per-query series, but very small N tightens the
   CI artificially. Interpret CIs at N<10 with suspicion.
