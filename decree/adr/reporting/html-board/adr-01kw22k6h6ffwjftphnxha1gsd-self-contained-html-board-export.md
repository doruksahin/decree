# ADR-01KW22K6H6FFWJFTPHNXHA1GSD Self-contained HTML Board Export

## Context and Problem Statement

The desired end state may use Astro or another frontend build system, but the
first slice needs a low-friction board export that proves the data model and
presentation without adding a JavaScript build pipeline to the Python package.

The board must be generated by the decree CLI and be easy to inspect locally.
It should not mutate project state or replace terminal commands.

## Decision Drivers

- The PoC should have no runtime server dependency.
- The generated artifact should be one file so it is easy to share.
- The implementation should stay inside the current Python packaging model.
- The data contract should remain explicit enough to later feed an Astro view.
- Query commands must not silently rebuild derived artifacts.

## Considered Options

- Add an Astro app and run a Node build from the CLI.
- Generate self-contained HTML from a bundled Jinja template.
- Generate raw JSON only and leave rendering to external tools.

## Decision Outcome

Chosen option: "Generate self-contained HTML from a bundled Jinja template",
because it validates the sprint board experience with minimal dependencies and
keeps the CLI output as a deterministic local artifact.

Astro remains a plausible next step after the board data shape is stable. If we
adopt it later, the CLI can keep producing the same board payload and render it
through a richer template pipeline.
