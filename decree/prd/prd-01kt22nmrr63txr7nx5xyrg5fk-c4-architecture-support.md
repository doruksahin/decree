# PRD-01KT22NMRR63TXR7NX5XYRG5FK C4 Architecture Support

## Problem Statement

Decree manages the lifecycle of SPECs, but SPECs describe system components — they have identity (name, tech stack), hierarchy (system > container > component), dependencies (A depends on B), and data contracts (A produces X, B consumes X). Today, this metadata is unvalidated. Users add C4 fields to spec frontmatter and use separate scripts to generate architecture diagrams. Decree ignores these fields entirely.

This means:
- A spec can claim `depends-on: ["nonexistent_component"]` and decree won't catch it
- A spec can consume an artifact that nothing produces — silent contract mismatch
- C4 diagrams are generated by a separate tool with a separate error format
- LLMs creating specs don't know C4 fields exist because decree's template doesn't include them

## Requirements

- Opt-in via `[types.spec.c4]` in decree.toml — non-C4 projects are unaffected
- Validate C4 field presence when enabled (id, c4_type, c4_name required)
- Validate c4_type against configured levels (system, container, component)
- Validate `parent` resolves to another spec's C4 id
- Validate `depends-on` entries resolve to other spec C4 ids
- Detect duplicate C4 ids across specs
- Filter dead/superseded specs out of C4 processing
- Generate C4Container Mermaid diagram via `decree graph`
- `decree new spec` scaffolds C4 fields when C4 is enabled
- All C4 code in isolated module (`c4.py`)

## Success Criteria

- `decree lint` catches: missing C4 fields, invalid c4_type, dangling parent, dangling depends-on, duplicate C4 ids
- `decree graph` produces valid C4Container Mermaid diagram that renders on GitHub
- Non-C4 projects: zero behavioral change, zero config required
- Existing 142 tests pass with no regressions

## Scope

**In scope (v1):** C4 hierarchy validation, depends-on validation, duplicate id detection, C4Container diagram generation, template scaffolding.

**Deferred (v2):** produces/consumes contract validation, data flow diagram, external system declarations, level-ordering enforcement in parent chains.

**Out of scope:** plugin architecture, multi-repo C4, C4 Level 4 (code diagrams).
