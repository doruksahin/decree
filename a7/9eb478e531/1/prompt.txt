  SPEC-X: carried over to SPRINT-2
  reason: API dependency not ready

  Bu sayede drift olmaz; çünkü sprint1 geçmişi silinmez, sprint2 de “bu iş nereden geldi?” sorusunu taşır.

  Ben üç ayrımı net koyardım:

  1. Aynı PRD altında yeni SPEC

     yeni iş doğdu.

     Bu carryover’dır. Sadece sprint kapanışında/rollover’da, explicit reason ile olmalı.

  3. Kapalı sprintten eski işi aylar sonra tekrar aktif yapmak
  PRD/ADR = context olabilir, sprintlere kalıcı olarak “ait” değildir.

• Aşağıdaki senaryoları force edersek drift riski düşük olur.

  Sprint Lifecycle

  - İlk kurulumda aktif sprint yoksa decree lint error verir.
  - decree sprint start S2 aktif sprint varken fail eder.
  - decree sprint rollover S2 aktif sprinti kapatır, S2’yi aktif yapar.
  - Aynı anda iki aktif sprint varsa lint error verir.
  - Kapalı sprinte normal komutla task eklenemez.

  Yeni İş

  - decree new spec default olarak aktif sprint’e eklenir.
  - Sprint dışı SPEC açmak istenirse explicit olmalı: --backlog --reason.
  - PRD/ADR context olarak sprint listesinde görünebilir ama task sayılmaz.
  - Asıl sprint task birimi çoğunlukla SPEC’tir.

  PRD Sprint1, SPEC Sprint2

  - PRD sprint1’de yazıldı.
  - Sprint2’de aynı PRD’ye bağlı yeni SPEC açıldı.
  - Bu geçerli. Çünkü yeni SPEC yeni iştir, carryover değildir.

  SPEC Carryover

  - SPEC sprint1’de başladı, bitmedi.
  - Sprint1 kapanırken sadece explicit carryover ile sprint2’ye geçebilir.
  - Reason zorunlu: blocked by API, scope split, dependency missing.
  - Sprint1 kapanışında “carried over to S2” kaydı kalır.
  - Sprint2’de aynı SPEC “carryover” olarak görünür.

  Rastgele Taşıma

  - Sprint ortasında SPEC’i S1’den S2’ye taşımak default fail eder.
  - Ancak özel override olabilir: move --force --reason, ama bunu mümkünse v1’de koymazdım.
  - Taşıma sadece rollover sırasında doğal olsun.

  Drop / Defer

  - Sprint kapanırken bitmeyen task için karar zorunlu:
      - carryover
      - defer-to-backlog
      - drop

  - Reason olmadan kapanış fail eder.
  - Dropped task tekrar açılacaksa yeni SPEC veya explicit reopen gerekir.

  Backlog

  - Backlog aktif sprint değildir.
  - Açık ama sprint’e alınmamış işler backlog’da durabilir.
  - Backlog item aktif sprint progress’ine girmez.
  - Sprint’e alınırken explicit sprint add SPEC-X gerekir.

  Eski Kapalı Sprint

  - Sadece audit/reopen gibi ağır ve gerekçeli komutlarla değişebilir.
  - Eski sprintteki iş aylar sonra tekrar gündeme gelirse tercihen yeni SPEC açılır.

  - decree progress default aktif sprint’i gösterir.
  - --sprint S1 belirli sprint’i gösterir.
  - --all-sprints her sprint’i gösterir.
  - --backlog backlog’u gösterir.
  - --corpus eski tüm doküman davranışını verir.

  - why, refs, lint, intent-check, health, index verify sprint filtresinden etkilenmez.
  - Çünkü eski sprintteki karar hâlâ bugünkü kodu yönetebilir.

  Lint Hataları

  - Aktif sprint yok.
  - Birden fazla aktif sprint var.
  - Incomplete sprint task kapanış sonucu olmadan kapalı sprintte kalmış.
  - Aynı SPEC iki sprintte carryover kaydı olmadan görünüyor.
  - Açık SPEC ne active sprint’te ne backlog’da.
  - Kapalı sprinte yeni task eklenmiş.

  Özet kural: PRD context’tir, SPEC iş birimidir. Yeni SPEC başka sprintte olabilir; aynı SPEC başka sprint’e sadece carryover olarak ve reason ile geçebilir.
